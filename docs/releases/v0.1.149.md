# Orca v0.1.149

Patch release: isolate realtime server item projection behind a shared
`RuntimeEventProjector` reducer so app-server streaming no longer keeps
assistant message, plan, reasoning, tool, file-change, and workflow lifecycle
state inside `ServerRequestWriter`.

## Changes

- **Shared event projector**: `RuntimeEventProjector` now reduces runtime JSONL
  event lines into typed `ServerEvent` values for assistant message, proposed
  plan, reasoning, command execution, MCP/dynamic tool call, file-change, and
  workflow item lifecycles.
- **Slim server writer**: `ServerRequestWriter` now buffers runtime output and
  serializes projected events instead of owning the projection state maps and
  event-specific dispatch methods itself.
- **Projection boundary coverage**: a focused projector test locks reasoning
  item start/delta/completion behavior, while the existing server writer and
  server-mode contracts continue to cover tool, file-change, workflow, plan,
  and app-server stream compatibility.
- **Roadmap alignment**: the production roadmap now records the shared runtime
  event projection boundary as the next Codex-style reducer slice.

## Compatibility

No changes to CLI flags, JSONL runtime event names, app-server wire event
shapes, thread-store projection output, command/exec protocol, shell session
protocol, workflow event payloads, MCP routing, external tool routing, approval
policy semantics, conversation storage, history storage, npm package layout, or
install commands.

This release is an internal architecture slice. Existing app-server clients
continue receiving the same item lifecycle and legacy protocol events while the
live stream projection logic moves into a reusable runtime reducer boundary.
