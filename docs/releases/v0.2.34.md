# Orca v0.2.34

Feature release: interrupt in-flight parallel read-only tool calls without
leaving detached work or delaying the next TUI prompt.

## Why This Release

The previous read-only batch scheduler lived in `orca-tools`, created scoped OS
threads, and checked cancellation only before spawning each request. Once a
slow MCP resource read or another concurrent read-only call had started, an
`Esc`, `Ctrl+C`, or `turn/interrupt` could not reach it. RuntimeHost had to wait
for that accidental worker owner before the interrupted turn could settle and
the next prompt could start.

This was a lifecycle ownership defect, not a missing cancellation check. Task
admission, concurrency, join order, panic classification, terminal selection,
publication, and persistence were split across the tool library and runtime.

## Changes

- **One invocation owner**: `RuntimeToolCallRuntime` owns each call's permit,
  cancellation view, started state, blocking task, join, panic classification,
  and exactly-once terminal.
- **Reliable interrupt and recovery**: parent cancellation reaches calls that
  have already started; RuntimeHost waits for every invocation to clean up
  before publishing the operation terminal or admitting the next prompt.
- **Truthful terminal races**: an observed natural completion wins a concurrent
  cancellation, while a panic after dispatch is recorded as indeterminate and
  is not presented as retry-safe.
- **Stable provider order**: calls still execute concurrently, but model replay,
  history, hooks, and public terminal rows retain provider request order.
- **Cancellable MCP resources**: resource list, template list, and resource read
  carry typed cancellation through the registry, client, and transport. stdio
  performs a bounded reconnect before returning from cancellation; SSE closes
  the request and stays reusable.
- **Direct-call compatibility**: CLI and workflow child-agent paths without an
  ambient Tokio runtime create a batch-owned runtime that is dropped after all
  workers join.
- **Old path deleted**: the `orca-tools` parallel batch helpers, pre-spawn-only
  cancellation test, top-level shim export, and read-only source-shape tests are
  removed rather than retained as a second scheduler.

## Compatibility

CLI arguments, TUI keys and flows, server methods and JSONL wrappers, persisted
records, public tool identities, DeepSeek request order, and
`max_read_parallel` behavior remain compatible. The user-visible change is
that interrupt now settles already-started parallel reads and returns the
thread to an idle state after cleanup.

P1.2a is the final ownership boundary for parallel read-only calls. Sequential
normal tools remain on their current path until P1.2b, and subagent calls retain
their child-agent runtime until P1.2c.

Pre-publish validation:

```bash
git diff --check
cargo test --workspace --all-targets -- --test-threads=1
cargo clippy --workspace --all-targets
npm --prefix site run build
npm --prefix site run check:seo
node scripts/release/test-stage-npm.mjs
```

Post-publish verification:

```bash
node scripts/release/verify-published.mjs --version 0.2.34 --repo echoVic/blade-deepseek --package @blade-ai/orca --bin orca
```

## Upgrade

Install or pin the release with:

```bash
npm install -g @blade-ai/orca@0.2.34
```

or:

```bash
curl -fsSL https://orcaagent.dev/install.sh | \
  INSTALL_DIR=/usr/local/bin ORCA_VERSION=0.2.34 sh
```
