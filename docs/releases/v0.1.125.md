# Orca v0.1.125

Patch release: continue the P0 runtime reducer split by moving
`RuntimeToolActorContext` out of `lifecycle.rs` and into a focused
`runtime_tool_actor` module.

## Changes

- **Dedicated tool actor context module**: `RuntimeToolActorContext` now lives
  in `runtime_tool_actor.rs`, where it owns approval, hook, user-input,
  normal-tool, permission-overlay, and active-task adapter methods.
- **Preserved public imports**: existing callers can continue importing
  `RuntimeToolActorContext` through `orca_runtime::lifecycle::*`; `lifecycle.rs`
  re-exports the moved context while keeping the lower-level task actor and
  turn-context primitives for follow-up slices.
- **Behavior-compatible execution**: normal-tool invocation, request-permission
  overlay merging, workflow IPC dispatch, task stop/list behavior, hook
  formatting, shell cancellation, and user-input result shaping are unchanged.
- **Documented P0 slice**: added a focused implementation plan for the tool
  actor context boundary so the next `RuntimeTaskActor` and server-state splits
  can continue from a smaller lifecycle surface.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, model selection behavior, task ids, turn numbering, approval response
wire shapes, or existing `orca_runtime::lifecycle::*` imports.

This release is an internal architecture slice. Runtime task lifecycle events,
tool events, shell task metadata, workflows, subagents, session persistence,
history projection, and server/TUI/headless turn execution remain unchanged.
