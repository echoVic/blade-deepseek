# Orca v0.1.151

Patch release: wire the runtime extension contributor kernel into normal tool
execution so future goal, memory, task, and tool lifecycle behavior can observe
real tool starts and finishes through a typed extension boundary.

## Changes

- **Tool execution extensions**: `ToolExecutionContext` can now carry an
  `ExtensionRegistry` plus thread and turn `ExtensionData` stores.
- **Lifecycle notifications**: `ToolExecutionActor` emits `on_tool_start`
  before normal tool dispatch and `on_tool_finish` after dispatch, including
  completed, blocked, aborted, and not-implemented outcomes.
- **Outcome mapping**: completed tools map to `Completed`, denied tools map to
  `Blocked`, dispatch errors map to `Aborted`, and not-implemented tools map to
  a failed outcome with `handler_executed: false`.
- **Regression coverage**: a focused runtime test proves a normal `read_file`
  tool execution notifies extension contributors with the expected thread,
  turn, call id, and completion outcome.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, TUI
rendering, server wire shapes, app-server events, JSONL event names, thread
history, goal storage, tool schemas, npm package layout, or install commands.

This remains an internal architecture slice. Existing callers that do not pass
an extension registry keep the previous behavior, while runtime code now has a
narrow path for extension-aware tool execution.

