# Orca v0.2.27 (Candidate)

Feature release: make TUI cancellation one-shot and operation-scoped so an
interrupted turn cannot be revived or leak cancellation into later work.

Status: pre-publish candidate. Focused cancellation, complete workspace,
Clippy, site, release-helper, and live-provider gates passed on 2026-07-15.
Remote Actions, GitHub Release, npm, and public-install verification remain
required before this note can be marked released.

## Why

The production TUI previously shared one mutable `CancelToken` across submitted
turns, manual compaction, Goal operations, and approved background
continuations. Entry points called `reset()` before starting new work. That
made cancellation a reusable flag owned by the application lifetime rather
than the operation lifetime: a later reset could clear an older interrupt, and
an older cancelled token could make a later operation inert.

This was a lifecycle boundary defect rather than a missing conditional. The
fix had to establish operation identity and one-shot ownership at every TUI
entry point, then remove the reset path instead of adding another special case.

## Changes

- **Typed operation identity**: `orca-core::cancel` now exposes
  `OperationId`, `OperationScope`, and `OperationCancellation`. Each scope owns
  a fresh token whose cancelled state is permanent.
- **Fresh scope per TUI operation**: submitted turns, mention submissions,
  workflow notifications, manual compaction, Goal set/resume/continuation, and
  approved background continuation start independent scopes.
- **Current-operation interrupts**: Esc and Ctrl+C cancel the controller's
  current scope. The agent loop receives the existing interrupt action without
  relying on a reusable token reset.
- **Old TUI path deleted**: production TUI code contains no
  `CancelToken::reset()` call and no second cancellation source of truth.
- **Behavioral proof**: an agent-loop test starts a delayed mock stream,
  cancels it through `OperationCancellation`, waits for the cancelled terminal
  event, submits another turn, and proves the new operation id produces output
  and completes successfully.

## Compatibility And Follow-Up

CLI arguments, TUI keys and interaction flows, server methods and JSONL events,
persisted records, provider behavior, and DeepSeek streaming, retry,
compaction, and tool behavior remain unchanged. The observable correction is
that cancellation applies only to the TUI operation active at interrupt time.

The server `turn/resume` processor still resets its active `CancelToken`. This
is an explicit follow-up boundary, not a permanent compatibility layer. It must
be removed when a runtime host/actor owns server turn generation, restarts an
interrupted turn with a fresh scope, and rejects stale completion, approval,
and continuation events.
