# Orca v0.1.87

Patch release: add blocked-host attribution to managed `command/exec` network
proxy diagnostics.

## Changes

- **Blocked host attribution**: managed proxy 403 responses now include a
  normalized `x-proxy-host` header when the target host is known.
- **Codex-style network permission groundwork**: clients can now distinguish
  both the block reason (`x-proxy-error`) and the blocked domain
  (`x-proxy-host`), which gives the next automatic ask-on-network-block flow a
  stable host to request.
- **Command/exec visibility**: server contract coverage proves curl-based
  `command/exec` calls surface the blocked host in stdout alongside the
  existing `blocked-by-allowlist` diagnostic.

## Compatibility

No changes to command/exec method names, permission profile TOML shape, sandbox
policy names, proxy environment variables, JSONL event names, TUI behavior, npm
package layout, or install commands.

Existing clients that only read `x-proxy-error` continue to work. Clients that
want to build automatic network permission prompts can now read
`x-proxy-host`.
