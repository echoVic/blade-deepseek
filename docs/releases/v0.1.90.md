# Orca v0.1.90

Patch release: make model-visible `bash` inherit active permission-profile
network policy and request network permission on eligible managed proxy blocks.

## Changes

- **Bash network policy inheritance**: model-visible `bash` tool calls now reuse
  the active permission profile's command sandbox resolution, including managed
  network domain policy, writable roots, denied roots, and Unix socket
  allowlists.
- **Bash permission request/retry**: eligible bash proxy blocks now emit a
  `permission_request` for the blocked host. When the client responds with a
  turn-scoped network `allow`, Orca retries the original bash command with the
  expanded proxy policy.
- **Denylist safety**: explicit denylist blocks remain final proxy diagnostics
  and are not escalated into permission requests.
- **Runtime bash boundary**: shell-session bash execution now lives behind a
  dedicated `runtime_bash` module instead of growing lifecycle orchestration,
  making the next tool-control boundary slice easier to isolate.

## Compatibility

No changes to tool names, permission profile TOML shape, command/exec wire
methods, proxy environment variables, TUI behavior, npm package layout, or
install commands.

Model-visible bash now follows the same managed network policy direction as
server `command/exec`, while preserving existing bash approval and result
projection behavior.
