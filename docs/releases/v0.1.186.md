# Orca v0.1.186

Patch release: route runtime `request_user_input` through the turn interaction
state boundary.

## Changes

- **User-input interaction state**: `RuntimeTurnInteractionState` now carries
  the runtime user-input handler alongside the permission-request handler.
- **Thread turn API**: `ThreadTurnRequest::with_user_input_handler` can install
  a per-turn `RuntimeUserInputHandler`, and controller turn setup forwards it
  into `AgentLoopContext`.
- **Provider/tool-cycle routing**: `runtime_turn_iteration`,
  `RuntimeProviderCycleInput`, `RuntimeStepSnapshot`, `RuntimeNormalToolTurnContext`,
  and `ToolExecutionContext` pass the user-input handler through the existing
  runtime request path.
- **Runtime special dispatch**: `RuntimeToolRouter` classifies
  `request_user_input` as a runtime special tool. With a handler, it executes
  through `runtime_user_input`; without one, it fails clearly with
  `request_user_input requires a runtime user input handler` instead of falling
  back to the normal tool placeholder.
- **Regression coverage**: focused tests verify grouped handler identity,
  special dispatch classification, direct tool execution, and a full
  `ThreadTurnRequest` mock-provider turn that answers `request_user_input`
  through the installed handler.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, server-mode wire events, or JSONL event names.

Existing TUI request-user-input behavior remains on the dedicated TUI
interaction adapter. Non-interactive runtime turns that do not install a
user-input handler now fail through the runtime special-tool path with a clear
handler-missing error.
