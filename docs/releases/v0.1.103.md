# Orca v0.1.103

Patch release: tighten the runtime turn loop's request-state boundaries before
the next tool-router refactor.

## Changes

- **Grouped turn iteration input**: `RuntimeTurnIterationStep` now takes one
  runtime input object instead of a long flat argument list, keeping the
  turn-loop step boundary explicit.
- **Grouped provider cycle input**: provider-cycle and provider-response
  handling now use focused input structs, reducing repeated plumbing between
  lifecycle and provider-turn code.
- **Request-scoped step context**: provider response handling and tool turns now
  share `RuntimeStepContext`, a focused request-state object for config, cwd,
  tool policy, approval policy, project instructions, memory, MCP registry,
  hooks, cancellation, task registry, workflow IPC, and permission handling.
- **Architecture guardrails**: the runtime lifecycle contract now prevents
  provider response and tool-turn code from drifting back to duplicated
  request-state argument lists.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, or model-visible tool behavior.

This release is an internal architecture slice inspired by Codex `StepContext`
and package-3-style `QueryConfig` / `QueryDeps` boundaries. It preserves
runtime behavior while making the next runtime tool-router refactor easier to
test.
