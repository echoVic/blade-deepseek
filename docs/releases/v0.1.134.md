# Orca v0.1.134

Patch release: harden the runtime test infrastructure so one panicking
ORCA_HOME-scoped test no longer turns later tests into misleading poisoned-lock
failures.

## Changes

- **Poison-tolerant test environment lock**: runtime tests now use
  `history::lock_test_env()` instead of locking `TEST_ENV_LOCK` directly with
  `expect("env lock")` or `unwrap()`.
- **Regression coverage**: a focused test intentionally poisons a local mutex and
  verifies the shared recovery path can acquire it afterward.
- **Shared helper adoption**: history, server, session, thread-store, and
  workflow-host tests now share the same lock recovery behavior for ORCA_HOME and
  node-path environment mutation.
- **Cleaner failure signal**: the original panic remains visible, but later tests
  should not cascade into fake `PoisonError` failures just because they need the
  same serialized environment guard.

## Compatibility

No changes to CLI flags, JSONL event names, bash tool input/output shape, shell
session protocol, command/exec wire shape, permission-profile configuration,
network allow/deny semantics, filesystem sandbox grants, Unix socket allowlist
semantics, MCP routing, external tool routing, conversation storage, history
storage, workflows, TUI rendering, npm package layout, or install commands.

This release is an internal reliability slice. Runtime behavior is unchanged; the
test suite now reports the first real panic more clearly instead of manufacturing
follow-on failures.

## Install

```bash
npm install -g @blade-ai/orca
orca --version
```

Direct binary install:

```bash
curl -fsSL https://orcaagent.dev/install.sh | sh
```

Pin a specific version:

```bash
curl -fsSL https://orcaagent.dev/install.sh | \
  INSTALL_DIR=/usr/local/bin ORCA_VERSION=0.1.134 sh
```
