# Orca v0.1.72

Patch release: permission profiles now expand bounded filesystem globs for read, write, read-write, and deny command sandbox entries.

## Changes

- **Permission-profile glob roots**: `[permission_profiles.<name>.filesystem]` entries such as `/workspace/docs/**/*.md = "read-write"` now expand to concrete sandbox roots before `command/exec` starts.
- **Split filesystem policy parity**: the behavior follows the Codex/package-3 direction of keeping permission profile semantics as typed policy while adapting glob entries to the active sandbox backend.
- **Safety boundary preserved**: broad globs without a static parent directory, such as `/*.env`, are still rejected instead of scanning from the filesystem root.

## Compatibility

No changes to CLI flags, config file syntax, server-mode method names, npm package layout, install commands, JSONL event names, public realtime event names, or TUI display behavior.

Existing literal `read`, `write`, `read-write`, and `deny` entries keep the same behavior. Existing bounded `deny` globs keep the same matching behavior with generalized error wording for broad glob rejection.
