# Orca v0.1.106

Patch release: make the normal-tool fallback path injectable and directly
testable while preserving CLI, TUI, server, workflow, permission, and
model-visible tool behavior.

## Changes

- **Injectable fallback boundary**: `RuntimeNormalToolExecutor` now delegates
  MCP, TOML external, and built-in fallback execution through
  `RuntimeNormalToolFallbackExecutor`.
- **Fallback context object**: `RuntimeNormalToolFallbackContext` groups the
  fallback request, roots, MCP registry, external tool definitions, truncation,
  shell timeout, and cancellation state.
- **Default behavior preserved**: the default fallback executor still calls the
  existing `orca_tools::execute_with_mcp_external_roots_policy_or_cancel` path.
- **Architecture guardrails**: the runtime lifecycle contract now checks that
  the fallback executor trait, context, and default executor remain in
  `runtime_normal_tool`.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, or model-visible tool behavior.

This release is another internal architecture slice inspired by Codex-style
request processor boundaries and package-3-style grouped runtime context
objects. It keeps the current execution behavior intact while making the
normal-tool fallback path easier to test, replace, and evolve toward
long-running shell/session and async subagent work.
