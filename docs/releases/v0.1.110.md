# Orca v0.1.110

Patch release: move historical projected tool completion rebuilding into the
shared tool-item projection module while preserving CLI, TUI, server, workflow,
history, and model-visible tool behavior.

## Changes

- **Shared completion projection**: `tool_item_projection::complete_projected_tool_item`
  now owns historical projected tool completion rebuilding.
- **Thread-store cleanup**: `thread_store/projection.rs` delegates completion to
  the shared helper instead of calling MCP, dynamic, commandExecution, and
  fileChange completed-item constructors directly.
- **Shape compatibility**: persisted commandExecution truncation metadata,
  dynamic-tool failure content, MCP structured-result parsing, and fileChange
  completion continue to project the same item shapes.
- **Architecture guardrails**: runtime ownership tests now reject regressions
  that move completed-item constructor calls back into thread-store projection.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, or model-visible tool behavior.

This release continues the Codex/package-3-inspired tool-system convergence
work by reducing the remaining realtime-vs-history item-shape ownership drift.
Persisted history and realtime streams now share one clearer projected-tool
completion path, leaving the next approval/result-shaping slices with less
thread-store-specific reconstruction logic to carry forward.
