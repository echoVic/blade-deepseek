# Orca v0.1.113

Patch release: group provider-to-tool-turn dispatch inputs behind
`RuntimeToolTurnsContext` while preserving CLI, TUI, server, tool, approval,
workflow, and history behavior.

## Changes

- **Grouped tool-turn dispatch context**: `run_tool_turns` now receives one
  `RuntimeToolTurnsContext` instead of a long provider response call list.
- **Provider handoff cleanup**: `RuntimeProviderResponseStep` constructs that
  context for the tool-call branch, keeping final assistant recording and tool
  request extraction unchanged.
- **Behavior compatibility**: runtime step context, event emission, sink
  output, conversation/history recording, cost tracking, background workflow
  state, and child executor handoff keep the same behavior.
- **Architecture guardrails**: runtime ownership tests now reject regressions
  that re-expand provider response handling into the old long `run_tool_turns`
  call list.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, approval prompts, or model-visible tool behavior.

This release continues the Codex/package-3-inspired tool-system convergence
work by shrinking the provider response to tool-turn dispatch boundary after
grouping normal tool-turn inputs.
