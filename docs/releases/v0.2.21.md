# Orca v0.2.21

Patch release: recover a DeepSeek turn that finishes without user-visible
content or a tool call, and preserve useful failure evidence when recovery is
not possible.

## Incident And Root Cause

A production Goal ran for about 59 minutes and completed its implementation,
verification, roadmap update, commit, and final `update_plan` tool call. The
tool result confirmed every plan item was complete, but the next DeepSeek turn
ended with:

```text
DeepSeek provider error: response did not contain content or tool calls
```

The released provider allowed one empty-response retry. Reaching this terminal
error therefore proves that the initial response and its retry both completed
without a non-empty assistant `content` field or any tool call. The historical
version did not persist raw SSE frames, so the saved log cannot distinguish an
empty delta, whitespace-only content, or a reasoning-only response.

The existing retry resent the same request unchanged. That protects history
integrity, but it can reproduce the same deterministic model behavior. Raising
the retry count would add latency and cost while also repeating transient
reasoning already displayed by the TUI.

## Changes

- **Semantic recovery request**: streaming and non-streaming DeepSeek paths keep
  one retry limit, but the retry now contains a request-local instruction to
  return visible assistant content or call an available tool instead of ending
  with reasoning alone.
- **Valid role ordering**: when the request ends in a user message, the recovery
  instruction is merged into that request-local message. After a tool result,
  it is appended as a new user message. The original `Conversation` remains
  unchanged and the synthetic instruction is never persisted.
- **Tool boundary preserved**: valid reasoning attached to the preceding tool
  call, the provider call id, and its matching tool result remain intact. The
  invalid response reasoning is neither replayed nor written to history.
- **No duplicate thinking**: if the failed streaming attempt already emitted
  reasoning, retry reasoning is suppressed from the live callback while
  recovered content and tool progress continue normally.
- **Complete provider usage**: when DeepSeek reports usage for both attempts,
  the recovered turn returns their saturating sum, including usage attached to
  a terminal recovery error. Foreground TUI and controller paths retain their
  normal cost and budget accounting instead of losing the failed attempts.
- **Budgeted foreground admission**: while `max_budget_usd` is active, TUI
  provider requests are admitted serially against the synchronized session
  ledger. A later foreground turn fails preflight before its prompt enters
  conversation history once completed usage is over budget. Synchronous child
  usage is merged, emitted, persisted, and checked immediately; parallel child
  batches are disabled and asynchronous child agents are rejected in this
  mode. Each admitted synchronous child receives only the aggregate budget
  remaining in its parent, and a child that crosses it stops later children in
  the same provider turn.
- **Resumable usage epochs**: a new `session.usage_baseline` record snapshots
  the aggregate when an existing transcript is resumed. Later foreground
  cumulative snapshots and detached task deltas are folded into the new epoch,
  preserving pre-resume usage without counting it twice.
- **Accurate Goal usage**: Goal token accounting now uses input plus output
  tokens. Cache hits remain diagnostic metadata because they are already a
  subset of input tokens. Detached completion and approved-continuation usage
  are applied as exact deltas, and Goal-store mutations are synchronized so
  concurrent foreground and background updates cannot overwrite one another.
- **Failure diagnostics**: foreground and background TUI tasks, plus
  controller-backed CLI and server turns, retain the provider error instead of
  the generic string `failed` or a status-only completion record.
- **Redacted persistence**: `session.completed` gains an optional, backward-
  compatible `error` field. Detached provider turns use the task-correlated
  `background_task.provider_response` record for status, redacted error, and
  priced usage. Their terminal transition is atomic with foreground-state
  observation, and they do not write a global `session.completed` record. This
  keeps a slow background turn from overwriting a newer global completion while
  preserving task-level evidence and transcript totals. Session JSONL and task
  history apply the existing secret redactor before writing diagnostic text to
  disk.
- **Release workflow maintenance**: checkout, artifact upload/download, and
  Node setup actions in the tag-driven release workflow use their v5 releases.

## User Outcome

A long-running Goal no longer loses its final report merely because DeepSeek
ends one response without a terminal payload. Orca makes one targeted recovery
attempt without accepting reasoning-only text as the answer or poisoning the
resumable transcript.

If the provider still fails, the TUI reports the error as before, Goal mode
pauses safely, and `~/.orca` keeps a redacted reason in both the session and
main-task records. This makes later incident analysis possible without relying
on a screenshot or persisting credentials.

## Compatibility

No changes to CLI flags, slash-command syntax, model selection, reasoning-effort
configuration, approvals, hooks, workflows, MCP configuration, npm package
layout, or valid conversation history. Without `max_budget_usd`, subagent
execution behavior is unchanged. With a budget configured, synchronous child
work is serialized and asynchronous subagents fail closed so usage can be
reconciled in the parent turn.

Older session files without `session.completed.error` or
`background_task.provider_response` or `session.usage_baseline` remain readable.
Readers ignore the task-correlated provider record for global completion status
and aggregate its usage into restored transcript totals. On resume, the latest
baseline begins a new accounting epoch while foreground cumulative usage and
detached deltas remain distinct on disk, avoiding loss or double counting. The
recovery instruction exists only in the outbound retry request and does not
change stored user messages or later replay.

Existing Goal records remain readable, but v0.2.21 does not rewrite historical
`tokens_used` values that may already include cache hits twice. New usage deltas
use input plus output tokens only.
