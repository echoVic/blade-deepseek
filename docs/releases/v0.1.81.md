# Orca v0.1.81

Patch release: make managed network proxy policy blocks easier to diagnose from `command/exec`.

## Changes

- **Structured proxy block reasons**: permission-profile network denials now return Codex-style `x-proxy-error` reasons for explicit denylist hits and allowlist misses.
- **Command exec visibility**: `command/exec` output now preserves those proxy headers, so server clients can distinguish `blocked-by-denylist` from `blocked-by-allowlist` without guessing from a generic `403`.
- **Architecture alignment**: runtime network policy now resolves to an internal allow/block decision with a reason, matching the Codex network proxy direction while keeping the existing loopback proxy surface.

## Compatibility

No changes to permission profile config syntax, command/exec method names, sandbox selection, proxy URL environment variables, CLI flags, JSONL event names, TUI behavior, npm package layout, or install commands.

Existing denied requests still receive HTTP `403`; the diagnostic header is now more specific.
