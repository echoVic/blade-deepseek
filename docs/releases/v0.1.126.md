# Orca v0.1.126

Patch release: continue the Codex-style server boundary split by moving
`command/exec` active process state out of `server.rs` and into a focused
`server/command_exec_manager.rs` module.

## Changes

- **Dedicated command exec manager module**: `CommandExecManager`,
  `CommandExecProcess`, and `CommandExecDrainOutcome` now live in
  `server/command_exec_manager.rs`, where they own active process records,
  duplicate process-id checks, streaming drain, stdin write, resize,
  termination, output-cap accounting, and drain outcomes for permission retry
  paths.
- **Smaller server orchestrator**: `server.rs` keeps high-level server state,
  command startup, sandbox resolution, permission prompting, and protocol
  orchestration while delegating active process bookkeeping to the manager.
- **Behavior-compatible command execution**: buffered output, streaming output
  deltas, stdin writes, PTY resize, process termination, output truncation,
  network permission retry, filesystem permission retry, and shutdown cleanup
  keep the same server wire shape.
- **Documented P0 slice**: added a focused implementation plan for the manager
  boundary so the next server active-turn and shell/session splits can continue
  from a smaller `server.rs`.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, model selection behavior, command/exec options, process id semantics,
output cap fields, sandbox policy fields, or permission response wire shapes.

This release is an internal architecture slice. Server-mode command execution,
thread/session state, shell sessions, workflows, subagents, history projection,
and server/TUI/headless turn execution remain unchanged.
