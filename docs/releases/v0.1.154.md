# Orca v0.1.154

Patch release: make goal terminal updates consume live runtime thread extension
state before accepting `complete` or `blocked` claims.

## Changes

- **Live goal terminal guard**: `update_goal` terminal status updates now require
  live thread extension state and at least one completed non-goal tool attempt
  observed by that thread.
- **Shared goal progress helper**: goal tool progress recording now has a public
  runtime helper used by both the extension lifecycle contributor and TUI
  runtime-adapter path.
- **TUI thread-state bridge**: `TuiConversationSession` exposes its
  `RuntimeThread` extension store, and TUI turns record completed normal tools
  into the same thread-scoped progress state before later `update_goal` calls.
- **Regression coverage**: tests prove terminal updates are rejected before live
  progress exists, accepted after recorded non-goal tool progress, and wired
  through the TUI runtime boundary.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, TUI display
layout, server wire shapes, app-server events, JSONL event names, thread
history, goal storage format, tool schemas, npm package layout, or install
commands.

This is an internal architecture slice that turns the v0.1.153 persistent
thread extension store into an active goal policy input. It narrows premature
terminal goal claims while preserving existing user-facing goal commands.
