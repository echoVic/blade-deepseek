# Orca v0.2.22

Feature release: connect `orca-file-search` to multi-root, multi-surface
discovery and unify files, Skills, Plugins, and MCP Resources/Resource
Templates into one atomic Mention system.

## Why

ADR-0001 gave Orca a streaming file-search engine, but the first cut still had
three product gaps: one search session covered only one workspace root, file
search was TUI-only, and visible `@name` text was resolved again at submission
time. That resolution-by-text approach becomes unsafe as soon as two roots
share a relative path, a Skill and a Plugin share a display name, or two MCP
servers expose Resources with the same name — the wrong object can be attached
even though the user selected the right row. See
[ADR-0002](../architecture/adr/0002-unified-atomic-mention-system.md) for the
full design.

## Changes

- **Multi-root file identity**: `orca-file-search::SearchSession::start_roots`
  accepts one or more canonicalized, deduplicated workspace roots. Every match
  carries its canonical owning root plus the root-relative path, so equal
  relative paths from different roots remain separate candidates and
  overlapping roots keep one identity per traversal. Exclude globs,
  Git-ignore behavior, result limits, cancellation, progress, and snapshot
  semantics apply across the whole multi-root session.
- **Two app-server search contracts**: the Codex-compatible
  `fuzzyFileSearch/sessionStart|Update|Stop` contract now accepts explicit
  multi-root input and streams file results with root identity. A new
  thread-bound `mention/search/start|update|stop` contract names a live
  `threadId` and discovers/expands candidates against that thread's own
  workspace roots and MCP registry, so a candidate found by the Mention
  session is guaranteed to validate and expand against the same runtime
  boundary it was discovered in.
- **Unified typed candidates**: Files, Skills, Plugins, MCP Resources, and
  Resource Templates share one `MentionCandidate`/`MentionTarget` model.
  Skills use the existing environment-aware loader per workspace root; Plugins
  are discovered from workspace `.orca/plugins`, `.codex/plugins`, and
  `$ORCA_HOME/plugins`; MCP Resources and Resource Templates come from
  `resources/list` and `resources/templates/list` on the session registry.
  Files remain a streaming catalog; the smaller Skill/Plugin/Resource catalog
  is fuzzy-scored with Nucleo and merged with the current file snapshot at
  projection time.
- **Atomic binding**: selecting a candidate inserts natural visible text into
  the composer and records a hidden `MentionBinding` pointing at the exact
  `MentionTarget`. Stable ids are opaque SHA-256 hashes of the fully typed,
  length-delimited target — never of display text alone — so equal names with
  different targets never collapse and component separators (`:`, path
  separators) cannot create an alias.
- **Edit-safe bindings**: bindings are reconciled on every composer edit. An
  edit entirely before a binding rebases its byte range; an edit entirely
  after leaves it unchanged; an edit overlapping the binding invalidates it;
  and a binding whose visible slice no longer matches the current text is
  discarded. Submission can never attach a previously selected object after
  its visible mention has been edited into something else.
- **Validated expansion**: TUI submission uses `SubmitWithMentions { prompt,
  bindings }`; app-server `turn/start` accepts structured `{"type":"mention",
  "name":..., "target":...}` input and builds the same binding model. Before a
  prompt enters model history, the runtime revalidates each bound target: bound
  files must still resolve inside the bound active workspace root, Skills must
  still match both id and canonical path from an active root, Plugin manifests
  must remain inside configured plugin roots with the bound manifest name, and
  MCP Resources are read through the same registry used during discovery.
  Duplicate targets are injected once.
- **Compatibility preserved**: unbound legacy `@file` mentions continue to use
  the existing workspace-safe parser, and explicit `$skill` injection is
  unchanged. Structured clients and TUI selections gain atomic identity
  without breaking older prompts or scripts.
- **Owned shutdown**: both search managers retain every reaper handle created
  by an individual `stop`; `stop_all` and `Drop` join those handles before the
  server returns, so late relay output can never be written after a session's
  stopped event.
