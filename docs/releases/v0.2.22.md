# Orca v0.2.22 (Draft)

Proposed patch release: bound process output and make child-process ownership
explicit across long-running Goal, tool, workflow, MCP, subagent, verifier, and
server workloads.

Status: planning draft. v0.2.22 is not released. The changes below describe the
candidate resource-governance scope and must be reconciled with the final code
and verification evidence before publication.

## Incident Boundary

The reported v0.2.18 Goal incident was not an Orca transcript heap leak. The
incident session and task records were about 1.00 MiB and 428 KiB on disk,
while a macOS Jetsam record showed 10 orphaned Node workers retaining 40.51 GiB
in total. The session repeatedly recorded Vitest/Tinypool `kill EPERM`; Orca's
native processes were about 27-31 MiB in the same Jetsam snapshot.

v0.2.19 fixed that direct cause by allowing a sandboxed process to signal its
own descendants in the macOS Seatbelt profiles. The planned v0.2.22 work is
separate defense in depth for additional resource-ownership, output, and
credential-handling gaps found while reviewing the broader runtime.

The full evidence and scope are documented in
[`docs/reports/2026-07-12-goal-resource-incident.md`](../reports/2026-07-12-goal-resource-incident.md).

## Candidate Changes

If integrated, the candidate would provide:

- **Bounded process ingress**: ordinary captured stdout and stderr retain at
  most 1 MiB per stream, with exact omitted-byte notices. Streaming shell,
  verifier, external-tool, and grep paths use the same bounded collector before
  later presentation truncation.
- **Truthful terminal output**: the TUI replaces a live streaming preview with
  the final canonical bounded output when a tool completes.
- **External tools**: a tool whose stdin write fails is terminated as a process
  tree and waited before Orca records the failure.
- **MCP and workflows**: stdio MCP servers and WorkflowHost children are
  terminated and reaped after timeout, cancellation, transport error, setup
  failure, or host drop. Workflow event handoff is bounded.
- **Async subagents**: locally launched workers have an owned process-group
  handle, stop reaps the worker tree, and natural exit is reaped. Recovered and
  legacy workers are signaled only after PID, process-group, and task identity
  checks, reducing PID-reuse risk.
- **Server shutdown**: stdin EOF and reader/writer failure trigger two-phase,
  bounded shutdown. Orca first allows normal turns to finish, then cancels
  remaining turns, closes interaction waiters, and terminates ordinary shell
  trees without changing a naturally completed shell into `Stopped`.
- **Credential handling**: internal async-worker API keys travel through a
  bounded anonymous stdin pipe. They no longer appear in process argv or
  persisted workflow child records.

## Expected Compatibility

The candidate scope is not expected to change CLI flags, slash commands,
approval modes, Goal records, history formats, provider configuration, MCP
configuration, server JSONL schemas, npm package names, or public APIs. The
target Unix behavior terminates owned process groups; non-Unix builds retain
direct-child kill and wait behavior. This compatibility statement must be
confirmed against the final integrated diff.

## Remaining Risk

- Windows does not yet guarantee descendant-tree termination beyond the direct
  child.
- The stdio MCP response channel remains unbounded even though each request is
  timeout-bounded and the child is reaped on terminal paths.
- WorkflowHost has per-step cleanup but no total workflow deadline or global
  event-count ceiling.
- The managed network proxy still uses one detached thread per accepted
  connection and needs explicit connection-count and I/O-deadline limits.

These are follow-up hardening areas; none matches the direct v0.2.18 incident
path.

## Required Verification

No verification is claimed by this draft. Before release, focused regressions
must cover stdin failure, output omission counts, inherited-pipe descendants,
MCP and workflow timeout/error/drop paths, owned and recovered async workers,
PID reuse, key handoff, server EOF, interaction waiter shutdown, ordinary shell
descendants, and natural terminal state.

The release candidate must also pass:

```bash
cargo fmt --all -- --check
git diff --check
cargo check --workspace --all-targets
cargo test --workspace --all-targets -- --test-threads=1
cargo clippy --workspace --all-targets
npm --prefix site run build
npm --prefix site run check:seo
node scripts/release/test-stage-npm.mjs
node scripts/release/test-verify-published.mjs
node scripts/release/test-real-api-e2e.mjs
node scripts/release/real-api-e2e.mjs --max-budget 0.02
```

Publication, post-publish verification, and upgrade instructions must be added
only after the candidate is integrated and the release artifacts exist.
