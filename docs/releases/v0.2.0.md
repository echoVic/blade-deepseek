# Orca v0.2.0

Feature release: clarify TUI permission approval dialogs with risk-specific
titles and keep `request_permissions` routed through the runtime permission
handler.

## Changes

- **Permission-specific approval titles**: runtime pending interaction records
  now preserve whether a permission request is for a network block, filesystem
  write grant, or unsandboxed shell retry.
- **TUI projection**: the runtime interaction adapter projects typed permission
  requests into a dedicated TUI approval event instead of flattening them into a
  generic approval prompt.
- **Approval modal readability**: the approval dialog now shows `Network
  Permission Required`, `Filesystem Permission Required`, or `Unsandboxed Shell
  Required` when the runtime can classify the requested permission. Normal tool
  approvals still show `Approval Required`.
- **Request-permissions reliability**: `request_permissions` is no longer
  intercepted by the normal tool approval gate before the runtime permission
  handler can review it. TUI and server users can approve permission requests,
  including session-scoped directory and network grants, through the intended
  pending-permission flow.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, server-mode wire event names, or JSONL event names.

This release improves the TUI user experience for permission prompts while
preserving the existing approve/deny flow and making `request_permissions`
permission grants consistently reach the same runtime permission semantics in
TUI and server flows.
