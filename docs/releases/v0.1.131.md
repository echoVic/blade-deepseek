# Orca v0.1.131

Patch release: continue the Codex/package-3-inspired runtime boundary split by
moving readonly tool-turn batch execution and recording out of `tool_turn.rs`
and into a focused `runtime_readonly_tool_turn.rs` module.

## Changes

- **Dedicated readonly tool-turn module**: readonly batch planning,
  hook-gated parallel execution, post-tool hook emission, batch result
  recording, and readonly tool-turn composition now live in
  `runtime_readonly_tool_turn.rs`.
- **Grouped readonly contexts**: readonly execution now receives
  `RuntimeReadonlyBatchContext` and `RuntimeReadonlyToolTurnContext`, keeping
  the new boundary from recreating another long argument surface.
- **Smaller tool-turn dispatcher**: `tool_turn.rs` keeps request cursoring,
  child-tool policy checks, subagent batch selection, readonly batch selection,
  and normal tool-turn execution while delegating readonly execution to the
  focused module.
- **Behavior-compatible flow**: readonly request grouping, hook blocking,
  result ordering, tool-result recording, event deltas, subagent batch
  selection, normal tool turns, and disallowed child-tool failures keep the
  same runtime behavior.
- **Release-gate cleanup**: `orca-tui` disables rustdoc doctests because the
  crate has no fenced doctest examples; default `cargo test` no longer waits on
  an empty TUI rustdoc pass.

## Compatibility

No changes to CLI flags, JSONL event names, readonly tool result shape,
tool-call ids, hook inputs/outputs, MCP registry routing, external tool routing,
conversation storage, history storage, subagent behavior, normal tool
execution, workflows, shell sessions, server protocol, TUI rendering, npm
package layout, or install commands. TUI unit and integration tests still run
through the normal workspace test commands.

This release is an internal architecture slice. Readonly batch execution,
readonly result recording, normal tool turns, subagent batches, and tool-turn
dispatch behavior remain unchanged.
