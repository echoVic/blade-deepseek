# Orca v0.2.14

Patch release: add server-mode MCP elicitation parity for remote and bridged
clients, plus a command/exec ownership fix for shell listing.

## Changes

- **Server MCP elicitation events**: server-mode turns now emit
  `mcp_elicitation_request` when an stdio MCP tool sends `elicitation/create`
  during a tool call.
- **Protocol response method**: clients can answer with
  `mcp_elicitation/respond` using the stable `requestId` from the request event,
  plus `accepted` and optional `contentJson`.
- **Request-id discipline**: server-mode pending MCP elicitation requests are
  scoped by active turn id before server and request names, reject duplicate ids
  without replacing the original waiter, and report unknown response ids clearly.
- **Runtime handler plumbing**: MCP elicitation handlers now travel through the
  same turn-scoped interaction capability path as permission and user-input
  handlers before reaching normal tool execution.
- **Interrupt-safe waiting**: `turn/interrupt` cancels a turn that is waiting on
  an unanswered server-mode MCP elicitation request, removes the pending request,
  and completes the turn as cancelled instead of waiting forever for the MCP
  server prompt response.
- **MCP transport cancellation**: cancel-aware stdio MCP calls now poll cancel
  state without consuming the notification-count guard, and SSE MCP calls with
  an elicitation handler use an owned worker path so cancellation returns
  promptly without requiring the handler to be thread-shared.
- **End-to-end continuation**: after a client accepts or declines the request,
  Orca writes the response back to the MCP server and continues the original MCP
  tool call.
- **Command/exec shell ownership**: `shell/list` no longer reaps or lists the
  backing shell sessions owned by server `command/exec`, so a finished
  `command/exec` process still emits its `command_exec_completed` event through
  the command/exec owner instead of being consumed by the generic shell path.

## TUI impact

Existing TUI MCP elicitation behavior is unchanged. The benefit is architectural:
server-mode and future bridge/app clients now use the same waiting interaction
model as the TUI, so MCP auth/device-code/form prompts no longer require a
TUI-only local path.

The command/exec ownership fix protects long-running server tasks that TUI or
bridge clients inspect through list operations. Generic shell listing cannot
steal completion output from the command/exec task-control path.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, approval options, or shortcut behavior.

This release intentionally extends the server-mode protocol with the
`mcp_elicitation/respond` method and the `mcp_elicitation_request` /
`mcp_elicitation_resolved` JSONL events. Existing server-mode methods and TUI
event shapes keep their current behavior.

## Upgrade

Install or pin the release with:

```bash
npm install -g @blade-ai/orca@0.2.14
```

or:

```bash
curl -fsSL https://orcaagent.dev/install.sh | \
  INSTALL_DIR=/usr/local/bin ORCA_VERSION=0.2.14 sh
```
