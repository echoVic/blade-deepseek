# Orca v0.1.89

Patch release: extend automatic network permission request and retry support
from single-shot `command/exec` calls to streaming `processId` command
processes.

## Changes

- **Streaming command/exec network permission flow**: `command/exec` calls with
  a `threadId`, `permissionProfile`, `processId`, and streaming output now
  convert eligible managed proxy blocks into a `permission_request` event
  instead of completing the process immediately.
- **Same processId retry**: after `permission/respond` returns `allow` with
  `scope: "session"`, Orca retries the original command using the same
  `processId`, emits a fresh `command_exec_started`, streams stdout/stderr
  deltas, and completes normally.
- **Process lifecycle cleanup**: command process draining now consumes finished
  process state before deciding whether a managed proxy block should become a
  permission request, avoiding premature `command_exec_completed` events.
- **Long-running process coverage**: streaming command processes retain their
  managed network proxy until the process exits or is terminated, so delayed
  network blocks can still become permission requests on the next server drain.
- **Single-shot compatibility**: the v0.1.88 single-shot retry path and denylist
  behavior remain unchanged.

## Compatibility

No changes to command/exec method names, permission profile TOML shape, sandbox
policy names, proxy environment variables, TUI behavior, npm package layout, or
install commands.

Eligible managed proxy blocks can now trigger automatic network permission
requests for both single-shot server `command/exec` calls and streaming
`processId` command processes with a `threadId`.

## Install

```bash
npm install -g @blade-ai/orca
orca --version
```

Direct binary install:

```bash
curl -fsSL https://orcaagent.dev/install.sh | sh
```

Pin a specific version:

```bash
curl -fsSL https://orcaagent.dev/install.sh | \
  INSTALL_DIR=/usr/local/bin ORCA_VERSION=0.1.89 sh
```
