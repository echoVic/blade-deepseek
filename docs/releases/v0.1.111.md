# Orca v0.1.111

Patch release: group the tool approval gate inputs behind
`ToolApprovalGateContext` while preserving CLI, TUI, server, tool, approval,
and history behavior.

## Changes

- **Grouped approval context**: `ToolExecutionActor::handle_approval` now
  receives one `ToolApprovalGateContext` instead of a long approval-gate
  argument list.
- **Execution handoff cleanup**: `ToolExecutionActor::execute` passes config,
  events, sink, request, invocation, policy, strict auto-review, and delta
  emission through that context.
- **Behavior compatibility**: approval allow/ask/deny decisions, strict
  auto-review re-prompting, approval events, denied-result shaping, and
  tool-call requested/completed emission keep the same external behavior.
- **Architecture guardrails**: runtime ownership tests now reject regressions
  that re-expand the approval gate into the old long argument list.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, approval prompts, or model-visible tool behavior.

This release continues the Codex/package-3-inspired tool-system convergence
work by shrinking the approval path call surface before deeper approval/result
shaping moves.


