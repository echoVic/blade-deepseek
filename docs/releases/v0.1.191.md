# Orca v0.1.191

Patch release: make the provider-response step consume the named response
handoff directly.

## Changes

- **Provider-response input consumption**: `RuntimeProviderResponseStep::handle`
  now receives `RuntimeProviderResponseInput` instead of flattening step
  context, sampling state, event/sink refs, conversation/history refs, cost
  tracking, and background workflow refs back into a long argument list.
- **Provider-response executor bundle**: child-agent, workflow-child, and
  batch-child executors now flow through `RuntimeProviderResponseExecutors`.
- **Cycle response handoff**: `RuntimeTurnProviderCycleStep::handle_response`
  passes the response input and executor bundle through to the response step as
  named boundaries.
- **Architecture coverage**: the provider-response source contract now locks the
  input-consuming signature so future response work keeps the grouped handoff.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, server-mode wire events, JSONL event names, or tool behavior.

This release is an internal runtime architecture slice. Final assistant-message
recording, automatic memory extraction after final responses, tool request
selection, normal/readonly/subagent/workflow tool dispatch, cost tracking, and
background workflow bookkeeping keep the same behavior while the
provider-response boundary moves closer to Codex-style step context handoffs and
Claude Code-style named execution contexts.
