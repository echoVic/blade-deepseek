# Orca v0.1.184

Patch release: introduce a named request-scoped runtime step snapshot.

## Changes

- **Request-scoped runtime snapshot**: `RuntimeStepSnapshot` now owns the stable
  runtime inputs for one provider/tool-dispatch request, including config, cwd,
  tool policy, approval policy, instruction/memory/MCP/hook services,
  cancellation, task registry, workflow IPC, and permission handling.
- **Narrower step context**: `RuntimeStepContext` now carries the snapshot plus
  the kernel-bound runtime extension context instead of exposing every
  request-scoped field as a parallel member.
- **Provider response routing**: provider final-response handling reads
  response-recording and auto-memory settings through the snapshot.
- **Tool dispatch routing**: tool dispatch splits `RuntimeStepContext` into the
  request snapshot and extension binding before routing normal, readonly,
  workflow, and subagent tool turns.
- **Architecture guardrail**: focused runtime tests prevent request-scoped
  fields from drifting back out of the named snapshot contract.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, server-mode wire events, or JSONL event names.

Runtime turn setup, provider response handling, tool dispatch, approval
overlays, workflow tools, subagent batching, goal progress, memory extraction,
and history recording remain behavior preserving.
