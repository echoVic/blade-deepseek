# Orca v0.1.163

Patch release: move grouped runtime extension-context composition into the
`RuntimeTurnState` boundary.

## Changes

- **State-owned extension context**: `RuntimeTurnState` now exposes
  `extension_context()` and an `extension_context_from_parts(...)` helper for
  composing the runtime extension registry plus thread/turn stores into one
  `RuntimeExtensionContext`.
- **Cleaner loop boundary**: `agent_loop` no longer imports or directly calls
  `RuntimeExtensionContext::new(...)` or `RuntimeExtensionStores::new(...)`.
  Extension-context construction routes through `RuntimeTurnState`.
- **Borrow-safe handoff**: `agent_loop` still destructures turn state to pass the
  mutable cost tracker into `RuntimeTurnLoopInput`, then uses the state-owned
  helper to build the grouped extension context from the same registry and
  scoped stores.
- **Lifecycle preservation**: provider turns, normal tool lifecycle
  contributors, and goal progress recording receive the same registry,
  thread-store, and turn-store references as v0.1.162.
- **Regression coverage**: tests now verify the runtime state accessor returns
  the same registry and scoped stores, and architecture tests reject direct
  extension-store reconstruction inside `agent_loop`.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, permission
request wire shapes, TUI display layout, server events, JSONL event names,
thread metadata, tool schemas, npm package layout, or install commands.

This is an internal architecture slice in the Codex/package-3 direction: the
runtime state boundary now owns another piece of extension context composition
without changing model-visible tool execution behavior.
