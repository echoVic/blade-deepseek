# Orca v0.1.27

Bug fix release: kill the DeepSeek-cache **compaction storm** and the **summary-state shape drift** that was hitting multi-process `--continue` resumes.

## Changes

- **Wire-equivalent compaction gating**: `needs_compaction` is now computed against the actual wire prompt the provider will send (messages + injected `[Summary baseline]` / `[Summary update N]` framing + volatile overlay glued to the last message + tool schema JSON), not just the in-memory `messages` token sum. Previously the local budget could pass while the real wire prompt was already over the limit, so the next turn re-fired compaction immediately.
- **Compaction hysteresis**: when compaction triggers at `effective_limit`, the kept window is now compressed down to `60%` of the limit (new `target_compaction_limit()`) so the next turn has real headroom before the next trigger. Tiny summary deltas (under `MIN_DELTA_RENDERED_TOKENS = 80` est. tokens) fall back to local truncation instead of churning the cache.
- **Extractive render for natural-language turns**: `render_summary_delta` now applies the same head/tail/byte-budget tier policy to large `User` (huge stdin pastes) and `Assistant.content` blobs that it already applied to tool outputs. Small natural-language turns are still kept verbatim.
- **Persist inherited `summary_state` on resume/fork**: the resume path in both `controller.rs` and the TUI `bridge.rs` now writes the inherited `summary_state` into the new transcript via `append_summary_state`, even when the current turn doesn't compact. Without this, a multi-process `--continue` chain (e.g. pipe-eval) would load the summary into memory but never persist it, so the next process to resume would lose the shape and re-trigger compaction.
