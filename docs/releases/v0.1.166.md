# Orca v0.1.166

Patch release: move turn-loop input construction behind a focused runtime
turn-loop entrypoint.

## Changes

- **Focused turn-loop entrypoint**: `runtime_turn_loop` now exposes
  `run_agent_turn_loop` for agent-loop entry, so the module that owns the turn
  loop also owns construction of the internal `RuntimeTurnLoopInput`.
- **Thinner agent_loop handoff**: production `agent_loop` now passes a
  `RuntimeAgentTurnLoopInput` launch object and no longer calls
  `RuntimeTurnLoopInput::new(...)` directly.
- **Next request-state runway**: the wide internal turn-loop input remains
  inside `runtime_turn_loop`, making the next Codex/package-3-style
  per-sampling request-state split less invasive.
- **Regression coverage**: the runtime architecture contract now rejects direct
  `RuntimeTurnLoopInput::new(...)` construction from production `agent_loop`
  and verifies that `runtime_turn_loop` exposes the focused entrypoint.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, permission
request wire shapes, TUI display layout, server events, JSONL event names,
thread metadata, tool schemas, npm package layout, or install commands.

This is an internal architecture slice. Runtime turn setup, directive-resolved
loop policy, provider turns, tool execution, compaction, workflows, and child
agent execution keep the same user-visible behavior as v0.1.165.
