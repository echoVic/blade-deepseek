# Orca v0.1.189

Patch release: group provider-response I/O refs behind a named runtime
handoff.

## Changes

- **Provider-response I/O bundle**: `RuntimeProviderResponseInput` now carries
  response execution refs through `RuntimeProviderResponseIo` instead of
  exposing events, sink, conversation, history writer, cost tracker, and
  background workflow refs as parallel fields.
- **Kernel assembly point**: `RuntimeTurnKernel` assembles the provider-response
  I/O bundle together with its existing step context and sampling state handoff.
- **Execution-boundary destructuring**: provider response handling destructures
  the bundle only when it enters the final-response or tool-turn execution
  boundary, keeping the intermediate response input smaller.
- **Architecture coverage**: a source-level regression test locks the boundary
  so future provider/tool routing work keeps the grouped response I/O context
  instead of reintroducing duplicate flat fields.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, server-mode wire events, JSONL event names, or tool behavior.

This release is an internal runtime architecture slice. Provider response
folding, final-message recording, tool request extraction, and tool dispatch
keep the same behavior while the provider-response handoff becomes smaller and
better aligned with Codex-style turn and step context boundaries.
