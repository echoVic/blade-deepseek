# Orca v0.1.78

Patch release: route server-mode long-lived agent state through `RuntimeThread`, continuing the Codex/package-3-style convergence toward one runtime-owned thread surface.

## Changes

- **Server RuntimeThread adoption**: `ServerThread` now stores `thread: RuntimeThread` instead of directly owning `InteractiveSession` and `RuntimeSessionLifecycle`.
- **Runtime request helpers**: `RuntimeThread` now exposes request and cancel-aware request execution helpers used by server-mode turns.
- **Resume path preservation**: server resume/fork paths use `RuntimeThread::resume_same_thread`, preserving existing transcript replay and writer behavior while moving ownership behind the runtime boundary.
- **Architecture guard**: a new runtime boundary test prevents `server_runtime` from directly assembling `InteractiveSession`, `RuntimeSessionLifecycle`, or `ThreadTurnExecutor`.
- **Roadmap alignment**: production roadmap and implementation plan now mark server-mode `ServerThread` as backed by `RuntimeThread`, with TUI/headless adoption still open.

## Compatibility

No changes to CLI flags, server-mode method names, JSON request/response shapes, realtime event names, npm package layout, direct binary install commands, thread storage formats, permission-profile semantics, resume/fork behavior, cancellation, or TUI behavior.

This release is an architecture slice only. It removes duplicated server runtime ownership while keeping existing server runtime contracts intact.
