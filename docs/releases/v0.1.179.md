# Orca v0.1.179

Patch release: bind provider-response step context through the runtime turn
kernel.

## Changes

- **Kernel-owned extension stores**: `RuntimeTurnKernel` now retains the same
  runtime extension stores used to construct its reducer.
- **Step-context binding boundary**: provider response handling now asks the
  kernel to bind `RuntimeStepContext` extensions instead of wiring registry and
  stores directly inside `provider_turn`.
- **Tighter turn-state owner**: sampling request state, reducer state, and
  provider-response extension context now share one named per-turn kernel
  boundary, preparing the next state-transition consolidation slices.
- **Architecture guardrail**: a focused runtime test prevents provider turns
  from bypassing the kernel when binding step-context extensions.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, server-mode wire events, or JSONL event names.

Provider response handling, tool dispatch, approval overlays, workflow tools,
subagent batching, goal progress, and history recording remain behavior
preserving.
