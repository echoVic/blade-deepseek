# Orca v0.1.74

Patch release: permission-profile network domain policy now has a runtime enforcement path for `command/exec`.

## Changes

- **Managed domain policy proxy**: `command/exec` starts a per-command loopback HTTP proxy when the selected permission profile includes `[network.domains]` entries.
- **Allow/deny enforcement**: allowed HTTP hosts are forwarded through the proxy; denied hosts return `403` with `x-proxy-error: blocked-by-policy`.
- **Profile-chain semantics**: child permission profiles override parent domain policy entries for the same host.
- **Command environment routing**: Orca injects `HTTP_PROXY`, `HTTPS_PROXY`, and `ALL_PROXY` for the command process unless the request explicitly overrides those variables.
- **Bounded proxy lifecycle**: the per-command proxy stops accepting new connections when the command scope ends.

Example:

```toml
[permission_profiles.docs-online]
extends = ":workspace"

[permission_profiles.docs-online.network]
enabled = true

[permission_profiles.docs-online.network.domains]
"api.example.com" = "allow"
"blocked.example.com" = "deny"
```

## Compatibility

No changes to CLI flags, server-mode method names, npm package layout, install commands, JSONL event names, public realtime event names, or TUI display behavior.

This release is the first managed-proxy slice. HTTP requests through proxy-aware clients are covered. Unix-socket policy, richer HTTPS MITM/method enforcement, SOCKS routing, and interactive network ask flows remain follow-up work.
