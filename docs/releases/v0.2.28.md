# Orca v0.2.28 (Candidate)

Feature release: make server turn interruption and resume generation-owned so
the TUI and app-server can cancel, resume, and reclaim DeepSeek work without
reset races or duplicate terminal events.

Status: pre-publish candidate. Focused server/runtime, complete workspace,
Clippy, site, release-helper, and live-provider gates passed before publication.
Remote Actions, GitHub Release, npm, and public-install verification remain
required before this note can be marked released.

## Why

The server previously kept one mutable `CancelToken` for an active turn and
implemented `turn/resume` by calling `reset()`. That token outlived the
execution that observed it, so resume raced the old worker's cancellation
checkpoint instead of starting a new execution. Permission, user-input, and MCP
waiters also had incomplete cancellation ownership, and pending interaction ids
were not fenced to the generation that created them.

This was a turn-host lifecycle defect rather than a missing cancellation check.
The fix gives one worker and one ownership record to each logical turn, then
runs interrupted work again with a fresh generation scope.

## Changes

- **Generation-owned cancellation**: `ActiveTurnManager` owns one
  `turnId -> ActiveTurnEntry` record containing the worker, control, steer
  handle, join ownership, command mailbox, and session-permission metadata.
  Every generation receives a fresh one-shot `CancelToken`; no server path can
  reset an older generation.
- **Reliable resume**: `turn/resume` is queued through a capacity-one typed
  mailbox and is applied only after the interrupted generation returns. The
  same public turn id is reused, the original user prompt is not appended a
  second time, and duplicate resume commands coalesce.
- **Cancellation-aware interactions**: permission, user-input, and MCP
  elicitation waiters race their generation token and remove themselves on
  cancellation. Resumed interaction request ids carry an internal generation
  suffix, and stale responses are rejected before reaching a later waiter.
- **Terminal-event fence**: a generation holds `session.completed` until its
  outcome is known. A generation replaced by resume drops its stale terminal
  record, so one logical turn publishes one final `turn_completed` event.
- **Behavioral and real-provider proof**: server contract tests cover fresh
  scopes, stale responses, waiter cleanup, steer fencing, and one terminal
  event. The real DeepSeek gate interrupts after the first stream delta,
  resumes the same turn id, and verifies one successful terminal event.

## Compatibility And Migration

CLI arguments, TUI flow, server method names, public `turnId` values, persisted
message and turn records, provider selection, and DeepSeek request behavior stay
compatible. The first generation keeps existing interaction request-id shapes;
resumed generations add `-generation-N` to the emitted request id. Clients must
echo the request id received in the interaction event rather than synthesize it.

Successful interrupt/resume control statuses remain unchanged. A resume sent
before the current generation is interrupted or after it closes now returns a
protocol error instead of an acknowledgement that cannot restart work.
