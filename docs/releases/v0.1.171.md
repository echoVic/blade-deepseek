# Orca v0.1.171

Patch release: fix shell sandbox recovery for GitHub credential prompts and move
task session persistence out of project worktrees.

## Changes

- **Pathless sandbox escalation**: `bash` now handles sandbox denials that do
  not include a filesystem path, such as macOS/GitHub HTTPS credential-helper
  failures (`could not read Username ... Operation not permitted`). Runtime
  tool execution, JSONL `command/exec`, streaming `command/exec`, and the TUI
  can request approval to re-run the command without the filesystem sandbox.
- **Narrow filesystem grants preserved**: `.git/index.lock` and other
  path-attributed denials still use the existing targeted write-root approval
  flow instead of broadening every sandbox denial to full shell access.
- **User-level task sessions**: shell/subagent/workflow task session records now
  persist under `ORCA_HOME/task-sessions`, falling back to
  `~/.orca/task-sessions`, instead of creating `.orca/task-sessions` in the
  active project.
- **Legacy migration**: existing project-local `.orca/task-sessions` records are
  migrated into the user-level task session store when a registry is opened for
  that project.

## Compatibility

No changes to CLI flags, slash commands, model/provider configuration, MCP
configuration, hook schema, workflow scripts, npm package layout, install
commands, or JSONL event names.

Permission responses can now include:

```json
{ "shell": { "unsandboxed": true } }
```

Existing file-system and network permission responses continue to work. Commands
that report a concrete denied path still use file-system grants; the new shell
permission path is reserved for sandbox denials that cannot be fixed by granting
a specific path.

## Install

```bash
npm install -g @blade-ai/orca
orca --version
```

Direct binary install:

```bash
curl -fsSL https://orcaagent.dev/install.sh | sh
```

Pin a specific version:

```bash
curl -fsSL https://orcaagent.dev/install.sh | \
  INSTALL_DIR=/usr/local/bin ORCA_VERSION=0.1.171 sh
```
