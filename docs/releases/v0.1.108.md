# Orca v0.1.108

Patch release: centralize normal-tool invocation behind one runtime helper while
preserving CLI, TUI, server, workflow, permission, shell-session, and
model-visible tool behavior.

## Changes

- **Single normal-tool invocation helper**: `runtime_normal_tool` now exposes
  `execute_runtime_normal_tool`, so lifecycle code no longer directly
  instantiates `RuntimeNormalToolExecutor`.
- **Lifecycle call surface reduced**: both `RuntimeTaskActor` and
  `RuntimeToolActorContext` now pass a grouped `RuntimeNormalToolExecutionContext`
  into the same normal-tool helper.
- **Behavior preserved**: bash shell-session execution, MCP fallback execution,
  TOML external tool fallback execution, output truncation, shell timeout,
  cancellation, and permission-overlay handling continue through the same
  executor branches.
- **Architecture guardrails**: the runtime lifecycle contract now rejects direct
  normal-tool executor construction from `lifecycle.rs`.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, or model-visible tool behavior.

This release continues the Codex/package-3-inspired architecture work by making
normal-tool invocation a single named boundary. It keeps the current execution
behavior intact while preparing the runtime for later long-running shell,
worktree automation, and async subagent slices.

## Install

```bash
npm install -g @blade-ai/orca
orca --version
```

Direct binary install:

```bash
curl -fsSL https://orcaagent.dev/install.sh | sh
```

Pin a specific version:

```bash
curl -fsSL https://orcaagent.dev/install.sh | \
  INSTALL_DIR=/usr/local/bin ORCA_VERSION=0.1.108 sh
```
