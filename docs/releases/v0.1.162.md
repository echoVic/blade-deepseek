# Orca v0.1.162

Patch release: move grouped runtime extension context up through the turn loop,
turn iteration, and provider-cycle inputs.

## Changes

- **Turn loop extension context**: `RuntimeTurnLoopInput` now carries one
  `RuntimeExtensionContext` instead of separate extension registry, thread-store,
  and turn-store fields.
- **Iteration/provider handoff**: `RuntimeTurnIterationInput` and
  `RuntimeProviderCycleInput` pass the same grouped context forward instead of
  rebuilding extension stores from parallel refs.
- **Single construction point**: `agent_loop` creates the grouped runtime
  extension context once from `RuntimeTurnState` owned stores, keeping the actual
  store ownership unchanged.
- **Lifecycle preservation**: provider turns still build `RuntimeStepContext`
  with the same registry and scoped stores, so normal tool lifecycle
  contributors and goal progress recording keep the same behavior.
- **Regression coverage**: architecture tests now reject parallel extension
  fields in the loop, iteration, and provider-cycle input boundaries.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, permission
request wire shapes, TUI display layout, server events, JSONL event names,
thread metadata, tool schemas, npm package layout, or install commands.

This is an internal architecture slice in the Codex/package-3 direction:
extension context ownership keeps moving upward through the runtime turn
pipeline without changing model-visible tool execution behavior.
