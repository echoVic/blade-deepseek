# Orca v0.1.153

Patch release: move thread-scoped extension data into `RuntimeThread` so
extension contributors can keep stable thread-level runtime state across
multiple turns while preserving fresh turn-local state for each turn.

## Changes

- **RuntimeThread-owned extension store**: `RuntimeThread` now creates and owns
  the thread-scoped `ExtensionData` store using the runtime thread id as its
  level id.
- **Fresh turn extension stores**: each `RuntimeThread` turn receives a new
  turn extension id such as `<thread-id>:turn-1`, while direct legacy
  `ThreadTurnExecutor` use keeps the existing default behavior.
- **Turn-state reuse boundary**: `RuntimeTurnState` can now be created with an
  existing thread extension store, and `AgentLoopContext` has an explicit
  `with_runtime_thread_extensions` path for runtime-owned threads.
- **Regression coverage**: tests prove thread-scoped typed extension data
  survives across multiple turn states and turn-scoped data does not leak into
  later turns.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, TUI
rendering, server wire shapes, app-server events, JSONL event names, thread
history, goal storage format, tool schemas, npm package layout, or install
commands.

This is an internal architecture slice that gives goal, memory, task, skill,
and future runtime contributors the stable thread-level state boundary needed
for deeper Codex/package-3-style runtime behavior.



