# Orca v0.1.159

Patch release: group permission-sensitive extension store references behind
`RuntimeExtensionStores`.

## Changes

- **Grouped extension stores**: added `RuntimeExtensionStores` in the extension
  module as the single value for paired thread/turn extension stores.
- **Reducer construction boundary**: added
  `RuntimeTurnReducer::from_extension_stores(...)` so reducer creation no longer
  requires permission-sensitive call sites to unpack parallel store references.
- **Tool context cleanup**: runtime bash, normal-tool invocation, and tool-router
  contexts now carry `extension_stores` instead of separate
  `thread_extensions` / `turn_extensions` fields.
- **Behavior preservation**: model-visible `request_permissions`, bash network
  and filesystem auto-escalation, router permission-overlay transfer, and direct
  runtime-tool actor compatibility keep the same request / retry / merge
  behavior.
- **Regression coverage**: architecture tests now require grouped extension
  stores in permission-sensitive tool contexts and reject the old parallel
  context fields.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, permission
request wire shapes, TUI display layout, server events, JSONL event names,
thread metadata, tool schemas, npm package layout, or install commands.

This is an internal architecture slice in the Codex/package-3 direction:
runtime-owned reducers continue to handle completed-tool goal progress, runtime
directives, and permission overlay mutation, while their extension-store input
surface is now grouped and easier to route.
