# Orca v0.1.161

Patch release: move runtime extension grouping up to the step and normal
tool-turn contexts.

## Changes

- **Grouped runtime extension context**: `RuntimeStepContext` now carries a
  single `RuntimeExtensionContext` instead of separate extension registry,
  thread-store, and turn-store fields.
- **Normal tool-turn handoff**: `RuntimeNormalToolTurnContext` receives that
  grouped context directly, so normal tool turns no longer reconstruct extension
  stores from three parallel refs before invoking `ToolExecutionContext`.
- **Provider boundary stays explicit**: provider turns still create
  `RuntimeExtensionStores` once from the turn state stores, then hand the grouped
  runtime extension context into the provider-response/tool-turn path.
- **Lifecycle preservation**: normal tool execution still sends the same thread
  and turn stores to tool lifecycle contributors, preserving goal progress and
  extension notifications.
- **Regression coverage**: architecture tests now reject parallel extension
  fields in both `RuntimeStepContext` and `RuntimeNormalToolTurnContext`.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, permission
request wire shapes, TUI display layout, server events, JSONL event names,
thread metadata, tool schemas, npm package layout, or install commands.

This is an internal architecture slice in the Codex/package-3 direction:
extension context ownership keeps moving upward through the runtime call graph
without changing model-visible tool execution behavior.
