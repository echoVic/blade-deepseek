# Orca v0.1.104

Patch release: move runtime tool invocation dispatch behind a focused router
boundary while preserving CLI, TUI, server, workflow, subagent, permission, and
model-visible tool behavior.

## Changes

- **Runtime tool router**: `RuntimeToolRouter` now owns dispatch from runtime
  tool requests to workflow drafts/actions, workflows, subagents, subagent
  status, task list/stop, permission requests, workflow IPC, and normal tools.
- **Grouped invocation context**: `RuntimeToolInvocationContext` carries the
  dispatch-time state for config, cwd, event sinks, instructions, memory, MCP,
  hooks, cost tracking, cancellation, task registry, workflows, permission
  overlay, and child executors.
- **Narrower tool execution actor**: `ToolExecutionActor` keeps invocation
  preparation, approval handling, pre/post-tool hooks, and result finalization
  instead of also owning the large runtime dispatch match.
- **Architecture guardrails**: the runtime lifecycle contract now checks that
  the focused router module owns the tool invocation dispatch boundary and that
  `ToolExecutionActor` no longer exposes the old `dispatch_tool` method.

## Compatibility

No changes to CLI flags, JSON protocol method names, server event names,
permission profile TOML shape, install commands, npm package layout, provider
requests, or model-visible tool behavior.

This release is an internal architecture slice inspired by Codex-style request
processor boundaries and package-3-style grouped runtime context objects. It
preserves behavior while making the next tool-system convergence work easier to
test and review.

## Install

```bash
npm install -g @blade-ai/orca
orca --version
```

Direct binary install:

```bash
curl -fsSL https://orcaagent.dev/install.sh | sh
```

Pin a specific version:

```bash
curl -fsSL https://orcaagent.dev/install.sh | \
  INSTALL_DIR=/usr/local/bin ORCA_VERSION=0.1.104 sh
```
