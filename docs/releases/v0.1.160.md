# Orca v0.1.160

Patch release: move grouped extension-store routing up to the normal tool
execution entrypoint.

## Changes

- **Tool execution grouped stores**: `ToolExecutionContext` now carries
  `RuntimeExtensionStores` directly instead of separate thread/turn extension
  references.
- **No local reconstruction**: `ToolExecutionActor` no longer rebuilds grouped
  stores from parallel refs before router dispatch.
- **Lifecycle preservation**: tool lifecycle contributor start/finish events
  still receive the same thread and turn stores, now read from the grouped
  store value.
- **Tool-turn handoff**: `run_normal_tool_turn` constructs
  `RuntimeExtensionStores` once when handing the runtime step context into
  normal tool execution.
- **Regression coverage**: architecture tests now reject raw parallel extension
  fields in `ToolExecutionContext` and require lifecycle notifications to read
  from grouped stores.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, permission
request wire shapes, TUI display layout, server events, JSONL event names,
thread metadata, tool schemas, npm package layout, or install commands.

This is an internal architecture slice in the Codex/package-3 direction:
extension-store ownership keeps moving upward through the runtime call graph
without changing model-visible tool execution behavior.
