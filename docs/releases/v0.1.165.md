# Orca v0.1.165

Patch release: move directive-derived turn-loop policy behind the
lifecycle-owned `RuntimeTurnLoopState` boundary.

## Changes

- **Directive-resolved loop state**: `RuntimeTurnLoopState` now exposes the
  directive-resolved policy surface for each turn-loop iteration.
- **No directive accessors in agent_loop**: production `agent_loop` no longer
  destructures `RuntimeTurnLoopState` and no longer reads directive tool policy,
  runtime system messages, or model override directly.
- **Iteration-owned runtime policy view**: lifecycle resolves tool policy,
  runtime system messages, model override, cost tracker, cancellation, task
  registry, and grouped extension context into `RuntimeTurnLoopIterationState`.
- **Turn-loop input cleanup**: `RuntimeTurnLoopInput` now consumes the grouped
  loop state instead of receiving raw runtime refs plus separately derived
  directive fields.
- **Regression coverage**: architecture tests now reject directive policy
  resolution in production `agent_loop` and verify that turn-loop iterations
  request the lifecycle-owned resolved state at the iteration boundary.

## Compatibility

No changes to CLI flags, slash commands, approval policy semantics, permission
request wire shapes, TUI display layout, server events, JSONL event names,
thread metadata, tool schemas, npm package layout, or install commands.

This is an internal architecture slice. Runtime directives still switch models,
replace allowed tools, and inject runtime system messages with the same
user-visible behavior as v0.1.164.
