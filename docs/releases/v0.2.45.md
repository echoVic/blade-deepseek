# Orca v0.2.45

Patch release: align approval modes with their sandbox boundaries.

## Changes

- **Autonomous workspace editing**: `auto-edit` now allows Shell, network,
  agent, and write actions to proceed without an approval interruption while
  they remain inside the workspace sandbox. Requests that extend the sandbox
  still use the runtime permission flow.
- **Explicit full access**: `full-auto` now resolves the default command
  sandbox to `danger-full-access`, including app-server and TUI Bash paths.
  It no longer reaches an unsandboxed-shell permission prompt because of the
  default sandbox boundary.
- **Trust boundary consistency**: default folder-trust downgrades no longer
  override an explicit `full-auto` selection, while explicit permission
  profiles and command sandbox policies remain authoritative.
- **Documentation and release metadata**: README, harness contract, site
  changelog, and runtime configuration summaries now describe the same mode
  semantics.

## Compatibility

CLI flags, slash-command names, server wire shapes, JSONL event names, tool
schemas, permission profiles, and npm package layout are unchanged. The
behavioral change is intentional: `auto-edit` is autonomous only within its
workspace sandbox, while `full-auto` is the explicit full-access mode.

## Verification

```bash
cargo fmt --all -- --check
cargo test -p orca-core -p orca-approval -p orca-runtime --lib
cargo clippy --workspace --all-targets
node scripts/release/test-stage-npm.mjs
node scripts/release/test-verify-published.mjs
npm --prefix site run build
npm --prefix site run check:seo
```

The release was also smoke-tested through the compiled app-server and Mock
provider: `auto-edit` writes inside the workspace without a human prompt and
requests permission outside it; `full-auto` writes outside the workspace
without a permission request.
