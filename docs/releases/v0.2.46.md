# Orca v0.2.46

Orca v0.2.46 fixes a Goal Mode control-plane ownership defect that could leave
an active goal in an unbounded automatic continuation loop after every
`update_goal` call failed.

## What Changed

- Goal tool schema visibility and runtime execution now share the same explicit
  `ThreadTurnToolMode::Goal` capability.
- `get_goal`, `create_goal`, and `update_goal` execute through runtime-special
  dispatch with the persistent session id and live extension stores. They never
  enter the readonly batch or `orca-normal-tool` worker.
- The TLS-backed `GOAL_HANDLER`, callback scope, and related tests are removed.
- Invalid Goal JSON and status transitions remain model-recoverable. Missing
  runtime ownership and GoalStore I/O failures record one failed result and stop
  the turn.
- A failed tracked Goal generation atomically changes an active goal to
  `stalled` without overwriting paused, blocked, complete, or budget-limited
  state.
- Pause, clear, complete, blocked, budget limit, runtime stall, and ordinary
  turns remove stale volatile Goal context.
- RuntimeHost rejects Goal tool exposure without a recorded session before the
  provider can observe the schema.

## Incident Evidence

The production session contained 463 requested and 463 failed `update_goal`
events across 120 automatic continuations. It left the goal active after
318,271,748 Goal-accounted tokens and 18,750 active seconds; the final session
usage estimate was $18.693824488. See
[`docs/reports/2026-07-18-goal-runtime-control-plane-incident.md`](../reports/2026-07-18-goal-runtime-control-plane-incident.md)
for the full evidence, regression sequence, thread-local explanation, and
reference comparison.

## Compatibility

Tool names, JSON schemas, slash commands, TUI keys, server methods, history
records, Goal database shape, npm package names, and native archive layout are
unchanged. Ordinary failed tools retain their existing model-recovery behavior.
